# **楽天商品監視ツールのBDDシナリオ**

**フィーチャー:** 設定ファイルに基づいて楽天市場の複数ページを監視し、指定時間内に商品の変更を通知する

このツールは、開発者がソースコードを直接編集することなく、設定ファイルの変更のみで監視対象や動作時間を柔軟に変更できることを目指す。
また、開発環境（Windows）と本番環境（Ubuntuサーバー）の違いを吸収するため、GitHub Actionsによる自動テストを品質保証の要とする。

### **背景: 全てのシナリオの前提条件**

\*\*前提:\*\*
  \* ツールと同じ階層に \`config.json\` のような設定ファイルが存在する。
  \* ツールと同じ階層に \`database.json\` のような、前回の監視結果を保存する状態ファイルが存在する。

### **シナリオ1: 設定ファイルに基づいて監視を実行する**

\*\*もし\*\* \`config.json\` に以下の情報が正しく設定されているならば:
  \* "urls": \["https://item.rakuten.co.jp/shop-a/product-1/", "https://item.rakuten.co.jp/shop-b/category-2/"\] (監視したいURLのリスト)
  \* "startTime": "08:00" (監視開始時刻)
  \* "endTime": "20:00" (監視終了時刻)
  \* "webhookUrl": "https://discord.com/api/webhooks/..." (DiscordのWebhook URL)
\*\*かつ\*\* 現在の時刻が "08:00" から "20:00" の間であるならば
\*\*とき\*\* 監視ツールがcronで実行されると
\*\*ならば\*\* 設定ファイルに記載された全てのURLに対して、商品情報のチェックが行われるべきである

### **シナリオ2: 監視時間外に実行された場合**

\*\*もし\*\* \`config.json\` の監視時間帯が "08:00" から "20:00" に設定されているならば
\*\*かつ\*\* 現在の時刻が "07:59" または "20:01" のように監視時間外であるならば
\*\*とき\*\* 監視ツールがcronで実行されると
\*\*ならば\*\* URLへのアクセスや商品チェックは一切行われず、静かに終了するべきである

### **シナリオ3: 新商品が追加された場合**

\*\*もし\*\* シナリオ1の条件が満たされており
\*\*かつ\*\* 監視対象URLのいずれかで、前回の監視時には存在しなかった「在庫あり」の新商品が発見されたならば
\*\*とき\*\* 監視ツールがページをチェックすると
\*\*ならば\*\* Discordに「【新商品】\[商品名\]が入荷しました！ \[価格\] \[商品URL\]」という形式で通知が送信されるべきである
\*\*かつ\*\* 新しい商品の情報が \`database.json\` に記録されるべきである

### **シナリオ4: 在庫切れ商品が再販された場合**

\*\*もし\*\* シナリオ1の条件が満たされており
\*\*かつ\*\* \`database.json\` に「売り切れ」として記録されていた商品が、今回の監視で「在庫あり」に変わったならば
\*\*とき\*\* 監視ツールがページをチェックすると
\*\*ならば\*\* Discordに「【再販】\[商品名\]の在庫が復活しました！ \[価格\] \[商品URL\]」という形式で通知が送信されるべきである
\*\*かつ\*\* \`database.json\` の該当商品の状態が「在庫あり」に更新されるべきである

### **シナリオ5: 安全な開発ワークフローと品質保証**

\*\*フィーチャー:\*\* AIによるコーディングを前提とした安全な開発と品質保証

\*\*シナリオ:\*\*
    \*\*もし\*\* 開発者がAI（ClaudeCode）に指示してコードを生成・変更し、その結果をGitHubにプッシュしたならば
    \*\*とき\*\* GitHub Actionsのワークフローが自動的にトリガーされると
    \*\*ならば\*\* 本番環境であるUbuntuとほぼ同じクリーンな仮想環境上で、全てのテストが実行されるべきである
    \*\*かつ\*\* テストが全て成功した場合のみ、そのコードは「本番環境で動作する可能性が高い」と判断されるべきである
    \*\*かつ\*\* 監視対象のURLや時間帯の変更は、ソースコードを編集するのではなく、サーバー上の \`config.json\` ファイルを修正することで行われるべきである
    \*\*かつ\*\* DiscordのWebhook URLのような機密情報は、ソースコードには含めず、GitHub Secretsを通じてテストされ、本番サーバーでは環境変数または設定ファイルから読み込まれるべきである

### **シナリオ6: ページの構造が変更された場合（Chaosテスト）**

\*\*もし\*\* 楽天市場のサイトリニューアルなどでページのHTML構造が変わり、ツールが商品情報を正しく解析できなくなったならば
\*\*とき\*\* 監視ツールがページをチェックすると
\*\*ならば\*\* \`LayoutChangeError\` が発生し、処理は中断されるべきである
\*\*かつ\*\* Discordに「警告: ページの構造が変更された可能性があります。ツールのメンテナンスが必要です。」という通知が送信されるべきである
\*\*かつ\*\* Prometheusの \`monitor\_fail\_total{type="layout"}\` メトリクスが1増加するべきである

### **シナリオ7: データベースに接続できない場合（Chaosテスト）**

\*\*もし\*\* データベースサーバーの障害などで、ツールがDBに接続できないならば
\*\*とき\*\* 監視ツールがデータベースへアクセスしようとすると
\*\*ならば\*\* 処理はリトライ（再試行）され、最終的に失敗した場合はエラーが記録されるべきである
\*\*かつ\*\* Discordに「重大なエラー: データベースに接続できません。システムを確認してください。」という障害通知が送信されるべきである
\*\*かつ\*\* Prometheusの \`monitor\_fail\_total{type="db"}\` メトリクスが1増加するべきである

### **シナリオ8: Discordへの通知が失敗する場合（Chaosテスト）**

\*\*もし\*\* DiscordのWebhook URLが無効になるか、Discord側で障害が発生しているならば
\*\*とき\*\* 監視ツールが通知を送信しようとすると
\*\*ならば\*\* 通知の送信はバックオフ（時間をおいて再試行）され、最終的に失敗した場合はエラーが記録されるべきである
\*\*かつ\*\* Prometheusの \`monitor\_fail\_total{type="discord"}\` メトリクスが1増加するべきである
